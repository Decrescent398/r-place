Git & GitHub for Beginners:

Wait… so what’s the difference between Git and GitHub?

Git is a version control system that helps you track changes in your code. Think of it as a time machine where you can:

✅ Save your progress at any time

✅ Revert back if things go wrong

✅ Work on multiple versions of your project

Imagine you're building a Lego spaceship. You keep adding new boosters, components, and complex instruments. But what if one day, your little sibling storms in and knocks the whole thing over? 😱 If you had taken photos of your spaceship at every stage, you could easily rebuild it step by step.

That’s what Git does for your code, it saves versions so you can always go back if something breaks!

GitHub = The Cloud for Your Code

GitHub is an online platform where you store your Git-tracked code, collaborate with others, and show off your projects!

Think of it like Google Drive, but built specifically for code. You and your friends (or teammates) can work on the same project without messing up each other's work. And best of all? It keeps everything backed up online!

Step 1: Create a GitHub Account

  1. Go to GitHub

  1. Click Sign Up

  1. Choose a cool username 😎, enter your email, and set a password.
  1. Verify your email and log in!

Congratulations, you now have access to the world's biggest open-source developer community!

Step 2: Installing Git

Windows: Download Git and install it.

After finishing the installation you will see this:

Mac: Open Terminal and type:

brew install git  # If using Homebrew

Linux: Open Terminal and type:

sudo apt install git  # Ubuntu/Debian

sudo dnf install git  # Fedora

Before starting things off, create a folder anywhere and open it in your preferred IDE. I'm going to use Vscode for this tutorial.

Okay, so once you’re in Vscode and within your recently created folder open the terminal by doing ctrl + ‘ in windows.

Step 3: Set Up Your Git Identity

Before you start using Git, tell it who you are:

git config --global user.name "Your github username"

git config --global user.email "your@email.com"

Why? Because Git records who made each change.

Step 4: Create Your First GitHub Repository

Go to Github and click on the profile icon

Then click on “Your Profile”

And then Repositories

Click on the + in the top right → New repository

Give it a name (e.g., awesome-project)

Choose Public or Private (Choose public for this tutorial because it simplifies things)

Click Create repository

Boom! 🎆 Your repository is live on GitHub! It’s an empty space now, but let’s fill it with some code.

Remember the spaceship from before? What you’ve done now is basically create a workstation with some basic parts like a instruction manual, some tape, snacks and a pen, for you to start building your lego spaceship

Step 5: Connect Your Local Project to GitHub

Now, go back to the terminal you previously opened and initialize Git in your project by using the command:

git init

This tells Git, "Hey, start tracking this folder!"

Link your GitHub repo to your local project:

git remote add origin https://github.com/your-username/awesome-project.git -> the link of your project

origin is the nickname for your GitHub repo!

For now, you can start coding and build your project

Step 6: Make Changes and Save Them

Check What's Changed

git status

This shows which files have changed!

Stage Your Changes

git add .  # To add all changed files to git

or

git add *.filextension  (Example: *.py adds all files with .py extension) # To add files of one type

Or

git add filename.extension (Example: main.py adds the main.py file ONLY)

Commit Your Changes

git commit -m "Your awesome commit message here!"

Commits are like saving your game progress! 🎮

Once you’ve finished a part of your project, (or, I just finished building my spaceship) but you’ve had an idea, what if I add a tailpiece to my rocket. Will it look better? How am I going to do this without messing up the spaceship I’ve built so far?

This is when you’re gonna use branching.

Step 7: Branching

Branching in GitHub (or Git) allows you to create separate versions of your project to work on new features, fixes, or experiments without affecting the main codebase.

Main branch: No Tailpiece

Feature branch: with tailpiece

To enter the default branch (or main version of your project):

git branch -M main

To create a new branch with an additional feature:

git checkout -b feature-branch

Once you’ve run this command you can continue coding the new feature this won’t affect your main branch (main instance of the project)

Switch Between Branches

git checkout main  ->  Switch to main branch to code on main branch

git checkout feature-branch  -> Switch back to feature branch to code on feature branch

IMPORTANT: make sure you commit changes before switching branches or merging them otherwise git won’t be able to merge these branches

Step 8: Merge Changes

If you like the feature you’ve added and want to include it as part of the main project

and you want to bring changes from a feature branch into main:

git checkout main  -> To switch to main branch

git merge feature-branch  -> To merge feature into main

Step 9: Upload (Push) Changes to GitHub

git push is a command used to upload your local code changes to GitHub (or any remote repository).

This is the equivalent of uploading a picture of the spaceship on instagram for all of you’re friends to see

git push -u origin main  # Push your main branch

(Point to remember: origin was the nickname we gave to our project at the beginning and main was the default branch you just merged changes into)

Since it’s your first time using git on your computer you will be prompted to sign in to give git access to your github page

Complete the necessary steps, and finally click on “Authorise git-ecosystem”

BOOM! And you’re done now you can open the repository page on github, refresh, and you will be able to see the file you have just pushed

Step 10: Forking a Repository 🍴

Forking lets you create your own copy of someone else's GitHub repo so you can make changes without affecting the original. To fork:

For this tutorial go to: https://github.com/Decrescent398/rplace-commits

Click the Fork button (top right).

And then “Create fork”

You now have your own version of the repo in your GitHub account!

The forked repository will also show up on your repositories tab

Step 11: Cloning a Repository

Cloning in GitHub means creating a copy of a remote repository (from GitHub) on your local machine.

So let’s say one of your friends at NASA has built a massive spaceship

What cloning does is basically create a copy of the repository on your PC so that you can add features

Once you’ve finished adding features push it back to the forked repository.

Here’s a step by step tutorial:

First create a new folder

Open the folder in Vscode

To clone a repository use:

git clone https://github.com/username/repository-name.git

For this tutorial clone the repository you forked from: https://github.com/Decrescent398/GithubTutorial

After cloning you will see something like this, open the “git-ninjas.txt file” and introduce yourself, make sure to write your name (or github username if you prefer), what you’re working on and the link to your github profile

Note to remember: The forked repository is the github page of your copy of the NASA spaceship with features you have just added and not the actual spaceship

IMPORTANT: Make sure you clone the forked repository and not the original one

Now, Navigate to your folder

And run the commands used in the previous steps

git branch -M main

git checkout -b main

git add .

Bonus: You can use git status to see all the changes you have made since we’ve only edited the git-ninjas.txt file we’ll get this output

git commit -m “your-name made changes”

And finally git push -u origin main

If you were paying attention all this while, you would have guessed by now, that these changes will now reflect on the github repository (or repo for short)

Step 12: Pull Requests - Contribute to Open Source

Alright, back to the NASA spaceship, what you’ve done now is create a your own clone of the NASA spaceship added some VERY useful features and pushed them to your github repository

You feel like these features could be very useful so you create a “Pull request” which is like saying “Hey NASA! I think these features would be awesome, can you include it in your Spaceship too pls pls pls??”.

Since you forked a project and made changes, you can request the original project owner to merge your changes!

  1. Go to your forked repo on GitHub.
  1. Click Pull Requests → New Pull Request.
  1. Select the original repo (https://github.com/Decrescent398/GithubTutorial) as the base and your fork as the compare.

4. Click Create Pull Request and explain your changes.

5. Wait for approval!

I’m going to get a notification that you’ve created a pull request

And within a few hours I’ll approve it

Check back in at https://github.com/Decrescent398/GithubTutorial, “git-ninjas.txt” and you should see your introduction in the original repository.

Additionally, If you’re the owner of a repository with multiple other collaborators, to get the changes that you’re friends have made in the repository you don’t need to keep cloning and forking every time, you can use the command git pull to directly get all the changes done while you were gone on your PC, and continue coding after

Alright so why does any of this actually matter?

Cloning

You're working in a company like Google.

The team lead pushes the latest code to GitHub.

You clone the repo locally to build a new feature or fix a bug.

Forking

You want to contribute to ReactJS or TensorFlow (open-source projects).

Since you don't have access to the original code, you fork the repo to your own GitHub account.

You make changes and then request the maintainers to merge them.

Pull Request (PR)

You're working with your team at Microsoft.

You create a new branch, add a new feature, and open a Pull Request for your teammates to review your code before merging it into the main branch.
